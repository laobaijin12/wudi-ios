# 系统级通知实现方案（防止 APP 退出后台收不到消息）

## 一、问题原因分析

### 1. 当前消息是怎么来的

- 新消息通过 **SyncWebSocketService** 的 **WebSocket** 从后端推送：
  - 连接：`ws(s)://host/ws_sync_api/ws?token=xxx`
  - 后端在有新消息时通过该 WebSocket 推 `messages` 等数据。
- 客户端在 **AppState.applySyncWSMessage** 里处理这些消息：
  - 更新未读数、会话缓存；
  - 调用 **updateAppBadgeAndNotify** → **NewMessageNotificationHelper.notify** 发**本地通知**（UNUserNotificationCenter）。

### 2. 为什么后台收不到

- iOS 对后台限制很严：
  - APP 进入后台后，**WebSocket 会被系统挂起或断开**，无法稳定维持长连接；
  - 没有“保持一条业务 WebSocket 常驻后台”的通用能力（除非是 VoIP 等特定类型）；
  - 因此**一旦退到后台或进程被挂起，就收不到 WebSocket 推送**，自然也就不会触发 `updateAppBadgeAndNotify`，本地通知不会发。
- 现有 **本地通知** 只有在「APP 进程还在、且能执行到 `NewMessageNotificationHelper.notify`」时才会出现，所以无法解决「完全后台/被杀后仍要收到新消息」的需求。

### 3. 结论

- 要**可靠地在 APP 退出后台（甚至被杀死）时仍能收到新消息**，不能只依赖 WebSocket + 本地通知。
- 必须使用 **Apple 提供的系统级通道：APNs（Apple Push Notification service）**，由**服务端**在有新消息时主动向 Apple 发起推送，再由系统把通知展示给用户。

---

## 二、方案概览：APNs 远程推送

- **流程简述**：
  1. 客户端向系统注册远程推送，拿到 **deviceToken**，并**上传给自家后端**（与用户/实例绑定）。
  2. 后端在有新消息时，除了走现有 WebSocket 推送（给在线客户端），再**调用 Apple APNs 接口**，把通知发到对应用户的 deviceToken。
  3. 即使用户未打开 APP 或 APP 已被杀死，**系统仍会接收 APNs 并显示通知**；用户点击通知可打开 APP 并跳转到对应会话（与现有本地通知的 `userInfo` 结构一致即可）。

- **与现有逻辑的关系**：
  - **前台 / 刚进后台且 WebSocket 仍可用**：继续用现有 WebSocket → `applySyncWSMessage` → `updateAppBadgeAndNotify` → 本地通知（或仅更新角标）；可选：同一消息可由服务端只发 APNs 给「未在该设备在线」的会话，避免重复。
  - **纯后台 / APP 被杀死**：只依赖 **APNs** 发系统通知；用户点击通知时，用与现有 **NewMessageNotificationDelegate** 相同的 `userInfo`（如 `instance_id_for_api`、`jid`、`display_name`）跳转会话。

## 三、实现步骤

### 1. 苹果侧配置（Apple Developer）

1. **App ID**  
   - 在 [Apple Developer](https://developer.apple.com) → Identifiers 中，为当前 App 的 Bundle ID 勾选 **Push Notifications** capability。

2. **APNs 密钥或证书**（二选一，推荐密钥）  
   - **推荐**：Certificates, Identifiers & Profiles → Keys → 创建 Key，勾选 **Apple Push Notifications service (APNs)**，下载 `.p8`，记下 **Key ID**；在 App ID 所在 Team 下记下 **Team ID**、**Bundle ID**；在 Keys 页可看到该 Key 的 **Key ID**。  
   - **或**：创建 APNs 证书（开发/生产），导出 `.p12`，供服务端使用。

3. **Provisioning Profile**  
   - 开发/发布描述文件中包含 **Push Notifications**，用 Xcode 自动管理或手动下载安装。

### 2. Xcode 工程配置

1. **Capability**  
   - 在 Xcode 中打开 WudiApp 工程 → Signing & Capabilities → 点击 **+ Capability** → 添加 **Push Notifications**（以及若用 Background Modes 可勾选 **Remote notifications**，便于收到推送后做静默刷新，非必须）。

2. **Background Modes（可选）**  
   - 若希望收到推送时有机会在后台执行一段代码（如预拉会话列表），可勾选 **Background Modes → Remote notifications**；若只做「收到即展示、点击再进 APP」则可不勾选。

### 3. 客户端代码（iOS）

需要做三件事：**注册远程推送、把 deviceToken 上传后端、在 APP 启动/恢复时用远程推送的 userInfo 跳转**。

#### 3.1 注册远程推送并上传 deviceToken

- 在 **AppDelegate** 或 **WudiAppApp** 里（若用纯 SwiftUI 无 AppDelegate，可用 `UIApplicationDelegateAdaptor` 把 AppDelegate 挂到 App 上）：
  - 在 `application(_:didFinishLaunchingWithOptions:)` 里请求通知权限（你已在 MainTabView 里请求过，可保留一处统一请求）；
  - 调用 `UIApplication.shared.registerForRemoteNotifications()`。
- 实现：
  - `application(_:didRegisterForRemoteNotificationsWithDeviceToken:)`：拿到 `deviceToken`，转为字符串，**调用后端接口上传**（与当前登录用户/实例绑定），例如 `POST /api/v1/device_token`，body：`{ "device_token": "xxx", "platform": "ios" }`。
  - `application(_:didFailToRegisterForRemoteNotificationsWithError:)`：打日志，便于排查（模拟器常失败，真机 + 正确 profile 才会成功）。

注意：**现有项目入口是纯 SwiftUI 的 `@main struct WudiAppApp`**，没有 AppDelegate，因此需要新增一个 **AppDelegate** 并用 `@UIApplicationDelegateAdaptor` 接入，在 AppDelegate 里实现上述注册与上传 deviceToken 的逻辑。

#### 3.2 收到远程推送时的处理（可选：静默/展示前修改）

- 若只做「系统默认展示 + 点击跳转」，可不实现；
- 若要在展示前改标题/内容或做静默拉取，可实现：
  - `application(_:didReceiveRemoteNotification:fetchCompletionHandler:)`（需开启 Remote notifications 的 Background Mode），在 completionHandler 里根据 payload 更新 UI 或角标。

#### 3.3 点击通知跳转会话（与现有逻辑统一）

- 远程推送的 **aps** 外可带自定义 **userInfo**，格式与现有本地通知一致即可，例如：
  - `instance_id_for_api`、`jid`、`display_name`
- **NewMessageNotificationDelegate** 的 `userNotificationCenter(_:didReceive:withCompletionHandler:)` 已经根据 `userInfo` 跳转；**只要服务端下发的 APNs payload 里带上相同结构的 userInfo**，用户点击远程推送时也会走同一套逻辑，无需改跳转代码。
- 若用户是**从「未启动」状态点击推送进入**，需保证在 APP 启动完成后再执行跳转（你已有 `pendingRoute` 机制，在 `bind(appState:)` 时处理，可继续沿用）。

### 4. 服务端需要做的事

1. **存储 deviceToken**  
   - 提供接口（如 `POST /api/v1/device_token`）接收客户端上传的 deviceToken，与 **user_id / 实例 / 设备** 绑定并落库。

2. **在新消息产生时发 APNs**  
   - 在现有「新消息」逻辑中（即当前通过 WebSocket 推送给前端的同一处），增加：  
     - 解析该消息属于哪个用户/哪些实例；  
     - 查该用户绑定的 iOS deviceToken（可多设备）；  
     - 调用 Apple APNs 接口（使用 .p8 或 .p12）向对应 deviceToken 发推送。  
   - Payload 建议与现有本地通知一致，例如：
     - `aps`: `alert`（title/subtitle/body）、`sound`、`badge`（可选）、`category` 为 `new-message`；
     - 自定义：`instance_id_for_api`、`jid`、`display_name`，便于客户端点击跳转。

3. **APNs 调用方式**  
   - 使用 .p8：基于 JWT 的 HTTP/2 API（推荐，无证书过期换证问题）；  
   - 或使用 .p12：传统 Binary 接口或 HTTP/2 均可。  
   - 开发阶段用 **sandbox** 环境，上架后用 **production**。

### 5. 与现有「本地通知」的配合

- **前台/WebSocket 仍连通时**：  
  - 现有逻辑：WebSocket → `applySyncWSMessage` → `updateAppBadgeAndNotify` → 本地通知。  
  - 可选策略：服务端对「已通过 WebSocket 成功推送过的设备」不再发 APNs，避免同一条消息在同一设备上出现两次通知；或统一由服务端只发 APNs，客户端收到 APNs 时更新角标并展示（需 Background Mode 且实现 `didReceiveRemoteNotification:fetchCompletionHandler:`）。  
  - 简单做法：**服务端对新消息一律发 APNs**；客户端若当时在前台且 WebSocket 已推送过，可仅更新角标、不再弹本地通知（例如在 `updateAppBadgeAndNotify` 里根据「是否由 WS 已展示过」做去重），避免重复。

- **后台/被杀**：  
  - 只依赖 APNs，无需改动现有 WebSocket/本地通知逻辑；点击通知后的跳转已由 `NewMessageNotificationDelegate` + `userInfo` 统一处理。

---

## 四、小结

| 项目           | 说明 |
|----------------|------|
| **问题根源**   | 后台时 WebSocket 不可靠，收不到新消息，导致无法触发本地通知。 |
| **解决方向**   | 使用 **APNs 远程推送**，由服务端在有新消息时主动推送到设备。 |
| **客户端必做** | 1）开启 Push Notifications capability；2）注册远程推送并上传 deviceToken；3）点击通知跳转沿用现有 `userInfo`。 |
| **服务端必做** | 1）接收并存储 deviceToken；2）新消息时调 APNs 接口推送，payload 含 `instance_id_for_api`、`jid`、`display_name`。 |
| **现有逻辑**   | 本地通知 + `NewMessageNotificationDelegate` 保留；远程推送使用相同 userInfo，实现「系统级通知 + 点击进会话」。 |

按上述步骤实现后，即使用户将 APP 切到后台或完全退出，只要有新消息，服务端就会通过 APNs 下发系统通知，用户即可在通知栏看到并点击进入对应会话。
